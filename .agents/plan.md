# LazyDex — Build Plan

> **Purpose**: Requirements document for AI agents building LazyDex. Organized by version — build one version at a time, in order. Each version is self-contained and testable.

---

## Project Overview

Local-only Android media tracker for tracking consumption across Novels, Manga, Anime, Games, Movies, and TV Shows. No accounts, no cloud, no social. Privacy-first.

**Core Features** (to be built incrementally):
1. Manual entry — URL scrape (novels) or AniList/MAL search (anime/manga) as auto-fill options inside the add form
   - v1.0: manual entry + URL scraping for novels
   - Post-v1.0: AniList/MyAnimeList search (metadata fetch, no login)
   - Future: full AniList/MAL sync (OAuth, two-way progress/status sync)
2. Track progress (simple Int current + optional Int total; unit label derived from category)
3. Toggle status: 5 statuses (Reading/Watching/Playing, Completed, On Hold, Dropped, Plan to) — display label adapts to category
4. View all items in a filterable list (by category and status)
5. Tap item → detail screen (title, cover, progress controls, status, notes, source URL button, edit/delete)
6. Edit and delete items — inline editing on detail screen (edit in place + save), delete with confirmation
7. Backup/restore data to JSON via SAF + auto-backup (scheduled, like Mihon's .mihonbk)
8. Merge or overwrite on import
9. Dark theme (default) + Light toggle in settings + dynamic color (Monet, Android 12+) + fallback palette (older Android) + Amoled mode option
10. Settings screen — Data (export, import), Theme (dark/light toggle, amoled mode), Backup (auto-backup schedule), About (version, credits, licenses)

**Non-Goals** (do not build in v1.0): User accounts, cloud sync, social features, multi-device, extensions/plugins, push notifications, widgets.
- Note: AniList/MyAnimeList sync (OAuth) is planned as a post-v1.0 feature, not in-scope for initial release.

```

├── LazyDexApp.kt              # Application class (Koin init)
├── MainActivity.kt             # Single Activity, sets ComposeContent
├── data/
│   ├── local/                  # Room DB, Entity, DAO
│   └── repository/             # MediaRepositoryImpl
├── domain/
│   ├── model/                  # MediaItem, MediaCategory (NOVEL, MANGA, ANIME, GAME, MOVIE, TV), UserStatus (pure Kotlin)
│   └── repository/             # MediaRepository interface
├── ui/
│   ├── theme/                  # Material3 theme, colors, typography
│   ├── navigation/             # NavGraph + route definitions
│   ├── home/                   # HomeScreen + HomeViewModel
│   ├── detail/                 # DetailScreen + DetailViewModel
│   ├── add/                    # AddItemScreen (bottom sheet) + AddItemViewModel
│   ├── settings/               # SettingsScreen + SettingsViewModel
│   └── components/             # Reusable composables (MediaCard, badges, chips, etc.)
├── di/Modules.kt               # Koin modules
├── scraper/MetadataScraper.kt  # OkHttp + Jsoup
├── util/UrlNormalizer.kt       # Canonical URL normalizer
├── backup/                     # BackupProcessor + BackupManager
└── res/                        # Strings, themes.xml, drawables
```

### Key Architectural Rules
1. Domain layer has zero Android dependencies. No Context, no Parcelable, no Android types.
2. ViewModels never reference Fragment/Activity types. No `context` in ViewModels.
3. Repository is the single source of truth. UI never reads Room directly.
4. Every write path goes through `MediaItem.normalize()` (progress capping, bounds enforcement).
5. Screens are stateless Composables — all state in ViewModels via StateFlow.
6. ID generation: `repository.add()` generates UUID. ViewModels never call `UUID.randomUUID()`.

## Database Schema

Single unified `media_items` table for all categories:

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT (UUID) | PK, generated by repository |
| `category` | TEXT | Enum: NOVEL, MANGA, ANIME, GAME, MOVIE, TV |
| `title` | TEXT | NOT NULL |
| `cover_url` | TEXT | nullable, scraped image URL |
| `source_url` | TEXT | nullable, the tracked URL |
| `status` | TEXT | READING, COMPLETED, ON_HOLD, DROPPED, PLAN_TO |
| `progress` | INTEGER | default 0, current chapter/episode/% |
| `total` | INTEGER | nullable, optional max |
| `notes` | TEXT | nullable, user notes |
| `created_at` | INTEGER | epoch millis |
| `updated_at` | INTEGER | epoch millis |

- Unit labels (Chapter/Episode/Volume/%) are derived from `category` in UI, not stored in DB
- Category-specific metadata (author, studio, developer) deferred to v1.1+ (nullable columns or JSON blob)

## Key Constraints & Non-Negotiables

1. **normalize() is mandatory on every write path** — repository calls it, never the ViewModel or UI.
2. **Atomic SQL for progress** — increment/decrement are single SQL UPDATE statements. No read-modify-write race. No debounce.
3. **DistinctUntilChanged on all Flow observations** — Room fires table-level invalidation. Repository applies `.distinctUntilChanged()` after domain mapping.
4. **One-shot events use Channel** — navigation, toasts, intents go through `Channel<HomeEvent>` / `receiveAsFlow()`, never StateFlow.
5. **Backup lastUpdated default is epoch 0 (0L)** — NOT `System.currentTimeMillis()`. Epoch 0 means local always wins merge conflicts against untimestamped imports.
6. **Backup schemaVersion missing defaults to 1** — never reject old backups.
7. **Single canonical UrlNormalizer** — used by both scraper and duplicate detection. No drift.
8. **No TypeConverters for enums** — store enum names as raw uppercase strings in Room columns.
9. **JUnit5 for unit tests, JUnit4 for Compose UI tests** — never mix. ComposeTestRule requires JUnit4 runtime.
10. **No WebView** — Jsoup only. JavaScript-rendered pages fail gracefully.
